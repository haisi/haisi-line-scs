This guide documents the `optimistic-locking` sample: a Spring Boot application that exists to
demonstrate, in working code, three ideas from the accompanying conference talk:

* Optimistic locking with JPA `@Version` -- and why the version has to live on the *aggregate
  root*, not on a child row.
* Mapping concurrency conflicts onto HTTP status codes and headers (`ETag`, `If-Match`,
  `If-None-Match`).
* Designing an idempotent HTTP API with the `Idempotency-Key` header.

The running example is a `Line`: an aggregate with a `LeftPoint` and a `RightPoint`, each of which
can be moved independently, subject to a couple of business invariants. Everything below is
generated from the code itself: the module structure comes from
https://spring.io/projects/spring-modulith[Spring Modulith]'s own analysis of the compiled
application, and every HTTP request/response pair comes from a passing
https://spring.io/projects/spring-restdocs[Spring REST Docs] test -- if the documentation and the
behaviour ever disagree, the build fails before either does.

The `Line` aggregate root carries the JPA `@Version` (`lockVersion`); its `LeftPoint` and
`RightPoint` children deliberately do **not** have one of their own. The consistency boundary is
the aggregate, not the row -- giving a child its own version would let two concurrent edits to
*different* points of the *same* line each succeed silently, even though together they might
violate an invariant that only the aggregate can see (e.g. the combined update budget below).
Because a move only ever touches a child row, Hibernate would not otherwise bump the root's
version at all; `LineRepository.findForUpdate` loads the aggregate with
`@Lock(OPTIMISTIC_FORCE_INCREMENT)` specifically to force that bump at commit.

[[domain-invariants]]
== Domain invariants

`Line.moveLeft`/`Line.moveRight` enforce two invariants before touching either child, and expose
the same checks -- one method per point x direction (`canMoveLeftPointLeft()`,
`canMoveLeftPointRight()`, `canMoveRightPointLeft()`, `canMoveRightPointRight()`) -- so the web
layer can reflect them as hypermedia affordances on each embedded point (see <<reading-a-line>>):

[cols="1,2,1", options="header"]
|===
|Invariant |Enforced in |Violation response

|The left point may never exceed the right point (checked on both create and move).
|`Line.moveLeft`, `Line.moveRight`, `Line` constructor
|`422 Unprocessable Content`

|The left point may never go below zero.
|`Line.moveLeft`
|`422 Unprocessable Content`

|At most 5 updates total, summed across *both* points, may ever be applied to a line.
|`Line.validateInvariants`
|`422 Unprocessable Content`
|===

Both example violations below are captured from real, passing tests: moving left when it is
already at zero, and moving with an exhausted update budget.

NOTE: Both invariants return the same `422`; the response body doesn't distinguish which one

in <<reading-a-line, GET's hypermedia links>>, is that once either invariant would be violated, the
corresponding embedded point's `moveLeft`/`moveRight` relation simply stops being offered.

== HTTP API

[[error-responses]]
=== Error responses

Every error this API returns -- whatever the status code, whatever raised it -- comes back as an
`application/problem+json` body (https://www.rfc-editor.org/rfc/rfc9457.html[RFC 9457], the
successor to RFC 7807), not the framework's classic `{"timestamp","status","error","path"}` shape.
That's `spring.mvc.problemdetails.enabled`, plus every domain exception in `line`/`shared`
(`LineNotFound`, `LineAlreadyExists`, `BusinessRuleViolated`, `IdempotencyKeyReused`,
`PreconditionRequired`, `StaleStateIdentified`) extending Spring's `ErrorResponseException` instead
of the older `@ResponseStatus`-on-a-plain-exception pattern -- see `ApiExceptionHandler` for the
single `@RestControllerAdvice` that ties framework-level failures (a malformed request body, an
unmapped route, ...) and these domain exceptions together into the same shape.

A 404 raised deep in `LineService` -- moving a line id that doesn't exist -- is as good an example
as any of the other status codes throughout this guide:

operation::move-left-not-found[snippets='http-request,http-response,response-fields']

`type` is conspicuously absent above: RFC 9457 defines it as defaulting to `about:blank` when a
problem has no more specific URI to identify it by, and this app never registers one, so Jackson
omits the field rather than writing the literal string. Every other example in this guide --
`create-line-conflict`, `move-left-below-zero`, `move-left-missing-precondition`,
`move-left-stale-precondition`, and so on -- carries this exact same envelope; only `status`,
`title`, and `detail` change.

==== Bean Validation errors

`CreateLineRequest.businessPartnerId` carries `@NotBlank`, and the controller parameter is
annotated `@Valid`, so a blank value is rejected before a create ever reaches `LineService`. Out of
the box, Spring's own handling of a failed `@Valid` collapses every constraint violation into one
deliberately generic `detail` -- `"Invalid request content."` -- which is RFC 9457-correct but
useless to a client that needs to know *which* field failed and why. `ApiExceptionHandler`
overrides `handleMethodArgumentNotValid` to add an `errors` array -- one `field`/`message` pair per
failed constraint -- as a ProblemDetail extension property, same envelope, more actionable body:

operation::create-line-validation-failed[snippets='http-request,http-response,response-fields']

Every field-level constraint this API declares -- not just `@NotBlank` -- surfaces the same way:
`MethodArgumentNotValidException.getBindingResult().getFieldErrors()` already carries one `FieldError`
per violation regardless of which annotation raised it, so a future `@NotEmpty`, `@Size`, or
`@Min`/`@Max` needs no extra handling here.

[[business-partner-scoping]]
=== Business-partner scoping

Every request below carries an `X-Partner-Id` header. This app is multi-tenant at the *token*
level, not the *request* level: a single bearer token can hold semantic roles for several business
partners at once (`line_#create@acme`, `line_#read@other-partner`, ...), so a request has to state
which one of those partners it's acting as -- `X-Partner-Id` is that statement, checked twice before
any domain logic runs:

. `BusinessPartnerFilter` rejects the request outright (`403`) if the named partner is one the
  caller holds *no* role for at all -- this runs for every request, regardless of which operation is
  being called.
. The endpoint's own `@PreAuthorize("hasRoleForPartner(...))` then checks for the *specific* role
  that operation needs (`line_#create`, `line_#read`, ...) against that same partner -- so being
  affiliated with a partner isn't enough if the caller only has, say, the read role and is
  attempting a create.

A `GET` additionally re-checks ownership against the line's own stored `businessPartnerId` once
it's loaded (`LineAuthorization.assertOwnership`), independent of the header -- so naming a partner
you're genuinely affiliated with doesn't let you read a *different* partner's line by id.

Omitting the header entirely is only valid for a caller whose role is business-partner-*independent*
(`hasRoleForAllPartners`) -- deleting a line, for instance, requires a user-role rather than a
business-partner role, and `DELETE /lines/{id}` doesn't accept the header at all. For every
business-partner-scoped operation, a missing header simply leaves `hasRoleForPartner` nothing to
check the role against, so it's denied by default -- there's no fallback that would otherwise leak
one partner's data to a caller who never named it. `GET /lines` makes this concrete: with the
header, results are filtered to that one partner; without it, only a business-partner-independent
caller can even reach that branch (every business-partner caller was already denied by
`@PreAuthorize`), and what they see is every partner's lines, unfiltered.

`BusinessPartnerFilter` and every `LineController` method above are annotated `@ADR(1)`, linking
straight back to the ADR embedded below, which records why a header was chosen over the alternatives.

include::../../../../docs/adr/0001-x-partner-id-header-for-business-partner-scoping.adoc[leveloffset=+2]

[[concurrency-control]]
== Non-functional concern: optimistic locking &amp; concurrency control

Every write that isn't a create is conditional: the client must state which version of the
resource it read, and the server rejects the write outright if that version is no longer current.
This app follows one decision rule throughout, matching the accompanying talk's cheat sheet:

[cols="2,1", options="header"]
|===
|Situation |HTTP status

|Concurrent writers collide and neither stated a precondition (`If-Match`).
|`409 Conflict`

|The client *did* state `If-Match`, but it no longer matches the current version.
|`412 Precondition Failed`

|A business rule/invariant would be violated by an otherwise well-formed request.
|`422 Unprocessable Content`

|`If-Match` is required for this write, but the client sent none.
|`428 Precondition Required`
|===

A move is rejected with `428` if `If-Match` is absent at all:

operation::move-left-missing-precondition[snippets='http-request,http-response']

...and with `412` if it names a version that is no longer current -- here, because a second writer's
move landed first:

operation::move-left-stale-precondition[snippets='http-request,http-response']

The `409` case is different in kind: it only ever happens when *nobody* stated a precondition, so
there is nothing to compare against a "current" version at request time -- the conflict is only
detectable at commit, via the forced-increment version CAS described in <<Architecture>>.
`LineControllerITTest.PutMove#concurrentMovesWithoutPrecondition_onlyOneSucceeds` exercises this
directly: eight requests race with no `If-Match` at all, exactly one succeeds, and every other one
resolves to either `409` or `412` depending on scheduling -- both are correct outcomes of the very
same race.

Every response that changes the resource's state -- `create` and any of the four point/direction
move endpoints -- carries a fresh `ETag`, so a client always has what it needs to build its next
`If-Match` without a separate `GET`.

== Non-functional concern: idempotency

`Idempotency-Key` (an IETF draft header, distinct from the RFC 9110 idempotency already built into
`PUT`) supplements a move's built-in optimistic-locking flow for callers whose retry logic can't
rely on tracking the server's version alone -- for example, a client that must retry after a
network timeout without knowing whether its first attempt actually landed.

Unlike the other concerns in this guide, this one isn't implemented in `LineController`/`LineService`
at all: `IdempotencyFilter` (see `shared.web`) is a single, generic `Filter` wired into the security
chain ahead of every controller. It only ever engages for an endpoint explicitly opted in via the
`@Idempotent` annotation -- today, only the four move endpoints carry it. An endpoint without the
annotation ignores `Idempotency-Key` entirely, even if a client sends it, exactly like
https://docs.stripe.com/api/idempotent_requests[Stripe's own idempotent-request API], which
explicitly excludes `GET`/`DELETE` ("idempotent by definition... has no effect"); see
`LineControllerITTest.IdempotencyFilterBehavior#getAndDelete_ignoreTheIdempotencyKeyHeaderEntirely`.
For an opted-in endpoint, the filter fingerprints the request (method, URI, `X-Partner-Id`, the
caller's subject, and body) and either replays a previously stored response, rejects a fingerprint
mismatch or an in-flight duplicate, or lets a fresh request through and stores what it produced --
the controller/service layer below never even knows a replay happened.

The gate order matters: a recognised `Idempotency-Key` is checked *before* `If-Match` is even
looked at (the filter runs before the request ever reaches the controller), so a retry replays the
original result even if its own `If-Match` has since gone stale:

operation::move-left-idempotent-replay[snippets='request-headers,http-request,http-response']

Reusing the same key with a *different* request body -- a different fingerprint for the same key --
is treated as a client error (`422`), not a replay; see
`LineControllerITTest.PutMove#reusingIdempotencyKeyWithDifferentBody_returns422`. Two concurrent
requests racing the very same fresh key is a different case again: exactly one wins the underlying
insert, and every other one gets `409 Conflict` immediately, without ever reaching `LineService` --
see `LineControllerITTest.IdempotencyFilterBehavior#concurrentRequestsWithSameFreshIdempotencyKey_onlyOneExecutes`.

Only a successful (`2xx`) response gets stored for replay. A `422`/`412`/other non-2xx outcome
releases the reservation instead of freezing that failure under the key forever, so a genuine retry
-- even one that fingerprints identically -- executes for real against whatever the current state is;
see `LineControllerITTest.IdempotencyFilterBehavior#nonSuccessResponseUnderIdempotencyKey_abandonsReservationSoRetryCanSucceed`.

Moving idempotency into a filter also moves its bookkeeping out of the business mutation's own
database transaction -- unlike an earlier version of this app, where the idempotency record was
written in the *same* transaction as the move, so a lost version CAS rolled both back together. A
crash between the business transaction committing and the filter recording that outcome can leave a
retry landing on an unexpected `412` instead of a clean replay; see
`LineControllerITTest.IdempotencyFilterBehavior#retryAfterCrashBetweenBusinessCommitAndComplete_landsOn412NotAReplay`
for exactly that scenario reproduced end-to-end, and ADR 2 below for the full trade-off (including
the alternatives considered and rejected):

include::../../../../docs/adr/0002-idempotency-reservation-commits-separately-from-the-business-transaction.adoc[leveloffset=+2]

A separate, `@SchedulerLock`-guarded (https://github.com/lukas-krecan/ShedLock[ShedLock]) cron job,
`IdempotencyHousekeeping`, deletes idempotency records older than 24 hours so the table doesn't grow
forever -- see `IdempotencyHousekeepingTest` for the executable proof, including that the underlying
lock is genuinely taken (not just that the cleanup query ran).

`IdempotencyService` also exposes three Micrometer metrics (scraped alongside every other metric
this app publishes -- see <<logging-and-tracing,the next section>> -- via `/actuator/prometheus`),
so an operator can see how the mechanism actually behaves in production rather than only in tests:

[cols="1,2", options="header"]
|===
|Metric |What it shows

|`idempotency_outcomes_total{outcome="..."}`
|A counter per `reserve()` outcome (`reserved`, `replay`, `fingerprint_mismatch`, `in_progress`).
`rate(idempotency_outcomes_total{outcome="replay"}[5m])` against the total rate answers "how often
is a replay actually serving a retry instead of new work".

|`idempotency_records`
|A gauge: the current row count in `idempotency_record` -- how large the table has grown since the
last hourly housekeeping sweep.

|`idempotency_reserve_duration_seconds`
|A timer around `reserve()` itself -- how fast the reservation lookup (and, for a fresh key, the
insert) is.
|===

See `LineControllerITTest.IdempotencyFilterBehavior#reserveOutcomesAreRecordedAsMetrics` for proof
these are genuinely updated during real request handling, not just declared.

[[logging-and-tracing]]
== Non-functional concern: logging &amp; tracing

Every request this app handles is logged and traced through three
https://jeap-admin-ch.github.io/docs/building-blocks/spring-boot-starters/jeap-spring-boot-starters/index.html[jEAP
Spring Boot starters], all pulled in transitively by `jeap-spring-boot-application-starter` and
`jeap-spring-boot-monitoring-starter` (see `pom.xml`) -- no custom logging code lives in this repo
at all:

* *`jeap-spring-boot-monitoring-starter`* wires up Micrometer Tracing, bridged to OpenTelemetry.
  Every inbound HTTP request becomes a span with its own `traceId`/`spanId`.
* *`jeap-spring-boot-logging-starter`* bridges that span into SLF4J's MDC, so every log line
  written while handling a request -- not just one summary line -- carries the same `traceId`,
  letting an operator grep one id and see every log line one request caused, including in whichever
  *other* jEAP services that request went on to call, since the id (and the `JEAP-APPLICATION-NAME`
  header identifying the caller) propagates across the call.
* *`jeap-spring-boot-rest-request-tracing`* additionally logs one summary line per request/response
  under the `ch.admin.bit.jeap.log.RestRequestTracer` logger -- method, URI, status, caller, user,
  and elapsed time -- at `DEBUG`, suppressed by default in production. `application.properties`
  turns it on (`logging.level.ch.admin.bit.jeap.log.RestRequestTracer=DEBUG`) so it's visible in
  this teaching repo; it also sets `management.tracing.sampling.probability=1.0` so *every* request
  is sampled rather than the 1-in-10 Spring Boot default, which would otherwise make the concept
  intermittently invisible at this app's low traffic volume.

A real, unmodified log line captured from a running instance of this app (console format; `cloudwatch`/
`rhos` platforms emit the same fields as JSON instead, via `jeap.logging.platform`):

[source]
----
2026-07-13 14:18:20,010 DEBUG [optimistic-locking,d1841ba0d612456731cbc08468cd8f3e,f6d9531aefc23854] c.a.bit.jeap.log.RestRequestTracer - PUT http://localhost:8080/lines/a1ce5faf-.../left/move-right result=200 caller=null user=demo-administrator
----

The bracketed triple after the app name is `[app,traceId,spanId]` -- the same two values available
in the MDC (keys `traceId`/`spanId`) to every other log statement emitted while that request was
being handled.

`LineControllerITTest.LoggingAndTracing` is the executable proof: it attaches a Logback
`ListAppender` directly to the `RestRequestTracer` logger around a real HTTP call through the full
filter chain, then asserts on what actually got logged --
`requestIsLoggedWithATraceIdAndASpanIdInMdc` checks the response line's `result=200` and that its
MDC carries a non-blank `traceId`/`spanId`; `unrelatedRequests_getDifferentTraceIds` checks that two
independent requests get two different ids, which is the whole point: a fixed, per-process id would
be worthless for distinguishing one request from another.
